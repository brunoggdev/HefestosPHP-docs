Pasta Config

Na pasta Config temos os arquivos de configuração do seu app. Esses arquivos normalmente retornam um array associativo com os valores desejados para cada configuração.

Você pode criar seus próprios arquivos de configuração aqui e acessá-los com a função auxiliar config().

Dito isso, o Hefestos conta com alguns arquivos pré-definidos utilizados para diversos propósitos pelo framework. São eles:

  • App
  • Database
  • Filtros
  • Rotas
  • Inicialização

App

Nesse arquivo você irá definir os valores de constantes que serão utilizados pelo framework para diversas ações e que também ficarão disponíveis em toda a sua aplicação, então atenção nesses dados. São eles:

  • AMBIENTE - Normalmente "desenvolvimento" ou "producao". Utilizado, por exemplo, para decidir se devemos ou não mostrar erros na tela.
  • URL_BASE - A url base do projeto, normalmente algo como "localhost:8080" em desenvolvimento ou "exemplo.com.br" em producao.
  • VERSAO_APP - Apenas indica a versão atual do app mas pode ser muito útil para coisas como "cache busting" ou logs.
  • TIMEZONE - O fuso horário padrão do seu projeto dentre os disponíveis no php.
  • MANUTENCAO - Apenas um valor booleano indicando se devemos suspender acesso ao app para casos de manutenção.

Database

Neste arquivo você irá confirar as informações de conexão com o banco de dados. Note que podem ser configuradas várias conexões com banco de dados, cada para um ambiente específico.

Dois tipos de banco de dados relacional são suportados nativamente: mysql ou sqlite. Caso opte por sqlite, basta informar o caminho completo do arquivo, caso contrário, é necessário informar host, nome do banco de dados, usuário e senha.

  • driver - O driver sql do php que deseja utilizar, no momento suportados "mysql" e "sqlite".
  • sqlite - O caminho completo do arquivo (ex: 'app/Database/sqlite/banco_de_dados.sqlite').
  • host - Host da conexão mysql, normalmente localhost se em desenvolvimento.
  • nome_db - Nome do banco de dados mysql configurado.
  • usuario - Usuario configurado no banco de dados mysql.
  • senha - Senha configurada no banco de dados mysql.

Filtros

Neste arquivo você irá configurar seus filtros de rota. Filtros de rota são classes com um único método aplicar no qual você define a lógica de cada filtro.

Para configurar, cada chave do array é o nome do filtro que utilizado nas rotas e o valor deve ser a classe (com namespace completo) que será utilizada nele.

Para aplicar dois filtros em sequência para um mesmo nome de filtro basta informar um array com ambas as classes na ordem desejada. 

use App\Filtros\FiltroUsuarioLogado;
use App\Filtros\FiltroUsuarioAdmin;

return [
    'logado' = FiltroUsuarioLogado::class,
    'admin' = [FiltroUsuarioLogado::class, FiltroUsuarioAdmin::class]
];

Neste exemplo, o filtro "logado" vai chamar a classe FiltroUsuarioLogado que irá lidar com o fato de o usuário estar ou não logado. Já o filtro "admin" irá tanto fazer isso quanto também a classe FiltroUsuarioAdmin para também checar se o usuário logado é admin.

Rotas

Aqui você pode configurar as rotas da sua aplicação. Isso pode ser feito através do objeto Rota que possui métodos estáticos para cada verbo http disponível (get, post, put, patch e delete).

Estes métodos recebem dois parâmetros, respectivamente: a url da rota e uma função responsável por processar a resposta, que deve sempre explicitamente retornar algo com return. 

use Hefestos\Rotas\Rota;

Rota::get('/', function () {
    return view('home');
});

Rota::post('/pet', function () {
    // Cadastrar pet

    return json([
        'status' => 'ok',
        'message' => 'Pet cadastrado com sucesso!'
        'data' => [...]
    ]);
});

Note que se uma requisição "get" for feita para "/registrar" ela não será reconhecida e receberá um erro 404 automaticamente.

Apesar de ser possível informar uma função para processar a requisicao diretamente na rota para coisas simples como no exemplo acima, normalmente você irá apenas apontar um método de um controller para processar à requisicao: 
// Uma string com o controller e seu respectivo metodo desejado separados por '::'
Rota::post('/pet', 'MeuController::meuMetodo');

// ou exatamente equivalente:

// Um array com o primeiro item sendo o namespace completo da classe e o segundo uma string com o metodo desejado
Rota::post('/pet', [MeuController::class, 'meuMetodo']);
Você também pode criar rotas dinâmicas, ou seja, partes entre as barras (/) da url que podem variar. Para tal basta colocar entre chaves a parte da rota que deseja, podendo dar o nome que achar melhor, e os valores recebidos serão repassados respectivamente como parâmetros para a função de resposta: 
// Responderá rotas como '/usuario/1' e '/usuario/56'
Rota::get('/usuario/{id_usuario}', 'ExemploController::metodo1');

// Responderá rotas como '/usuario/1/pets/3' e '/usuario/56/pets/1'
Rota::get('/usuario/{id_usuario}/pets/{id_pet}', [ExemploController::class, 'metodo2']);

Finalmente, você pode utilizar nas suas rotas os filtros mencionados no tópico anterior, também conhecidos como "middleware", permitindo um maior controle elas para, por exemplo, filtrar rotas acessíveis apenas por usuários logados.

Para tal, você utilizar o método filtro na rota desejada: 

// Será aplicado o(s) filtro(s) para a chave 'logado' na config de filtros. 
Rota::get('/uma-rota-para-comuns-logados', 'MeuController::meuMetodo')->filtro('logado');

// Na chave 'admin' podemos, por exemplo, combinar o filtro 'logado' com o 'admin' para que ambos sejam aplicados.
Rota::get('/uma-rota-para-admins-logados', 'MeuController::meuMetodo')->filtro('admin');

Você pode ainda agrupar várias rotas sob um único filtro utilizando o método agrupar(), informando o nome do filtro desejado e uma função onde pode declarar as rotas do grupo. Por exemplo: 

// Será aplicado o filtro 'admin' para todas as rotas declaradas dentro da função
Rota::agrupar('admin', function () {
    Rota::get('/rota-admin-1', 'MeuController::meuMetodo1');
    Rota::post('/rota-admin-2', 'MeuController::meuMetodo2');
    Rota::delete('/rota-admin-3', 'MeuController::meuMetodo3');
});

Você pode, claro, chamar ainda o metodo filtro individualmente nas rotas do grupo, mas tenha em mente que tomarão precedência sobre o filtro definido no grupo.

Inicialização

Este é um arquivo bem opcional, então não se preocupe caso não entenda bem seu uso no começo.

Ele é importado logo cedo na execução do app, tornando-o o lugar perfeito para quaisquer configuracões adicionais que você possa vir a desejar, como para alguma biblioteca ou etc.

Por exemplo, podemos usar esse arquivo para definir como queremos que o Hefestos lide com exceções específicas: 

use Hefestos\Core\App;

App::tratarExcecao(InvalidArgumentException::class, function ($ex) {
    // invés de apenas gerar um erro 500 diretamente quando uma InvalidArgumentException for lançada, 
    // você têm a opção de tratar essa exceção do seu jeito e retornar uma mensagem de erro personalizada

    // realizar logs, enviar emails, etc

    return view('erros/meu-erro-customizado', ['erro' => $ex]);
});

Atenção: usando esse método, é de sua responsabilidade fazer quaisquer logs ou definir código http da resposta e etc, uma vez que não será tratado diretamente pelo framework.
