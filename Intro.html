Introdução

HefestosPHP é um framework MVC desenvolvido com PHP moderno. Seu principal intuito é simplificar o desenvolvimento web com PHP para iniciantes, de modo a estruturar o código e reduzir a curva de aprendizado dos grande frameworks do mercado como Laravel, CodeIgniter ou Symphony.

No entanto, por tratar de conceitos de desenvolvimento web de maneira generalista, permite que o conhecimento adquirido com seu uso seja facilmente reciclado também em qualquer outra linguagem com arquitetura similar.

Instalação e utilização

O framework pode ser obtido no repositório do github. Você pode simplesmente baixar o arquivo zip ou ainda usar a funcionalidade de "template" do github para criar um clone em seu próprio repositório.

Ademais, a sessão "Utilização geral" no menu lateral está organizada de acordo com a estrutura de diretórios do Framework, mais especificamente da pasta app. Isto é, cada item refletido nessa sessão do menu corresponde à uma pasta ou arquivo dentro da pasta app do framework, onde você fará a maior parte do sistema.

No entanto, é recomendado que entenda antes como funciona o ciclo de vida de uma requisição no HefestosPHP, como exemplificado logo abaixo.

Ciclo de vida de uma requisição

Assim como em diversos frameworks, no PHP ou não, o ciclo de vida de uma requisição no HefestosPHP segue um padrão comum guiado pela arquitetura MVC:

Um roteador é utilizado para decidir qual controller responderá à uma requisição específica, e middlewares podem ser utilizados para filtrar essas requisições antes de atingirem o controller, sendo úteis para tarefas como controle de acesso por login, entre outros.

O controller, por sua vez, não precisa saber dos detalhes de como a model recupera as informações do banco de dados ou como a view implementa seu HTML.

Já o acesso ao banco de dados é de responsabilidade única da model e esta não se preocupa com qual controller a invocou ou qual view receberá os seus dados. Além disso, é utilizado um query builder para realizar as consultas ao banco de dados de forma mais prática.

Por fim, a view não se preocupa em saber de onde vêm os dados que necessita ou quem a invoca. Esta apenas organiza o HTML com os dados recebidos e devolve a resposta que será enviada ao cliente.
